# ProcurePing 额外需求 2.0

> 文档性质：现有 V1 需求的增量补充  
> 适用范围：`backend`、`pc-frontend`、`h5-frontend`、`admin-frontend`  
> 目标：把当前原型/MVP 补齐到更接近企业级反向电商/采购撮合系统的产品规格

---

## 1. 背景与当前差距

当前项目已经具备反向采购撮合平台的基础雏形：

- 后端使用 FastAPI，已有用户、公司、分支、分类、目录、需求、报价、订单、交付、纠纷、消息、通知、审计、模拟托管、部分企业后台 API。
- PC 和 H5 前端已经有 buyer/supplier/admin 的多页面结构。
- 独立 `admin-frontend` 已经接入部分真实后端接口，覆盖用户、公司、审核、订单、托管、支付事件、风险、通知、设置、审计等模块。

但距离企业级完整系统仍有明显差距：

- 前后端 API 路径、状态枚举、类型定义不一致，PC/H5/admin 三端不能完全稳定复用后端契约。
- `pc-frontend/pages/admin/*` 多数仍偏静态演示页，`admin-frontend` 更接近真实后台，两个后台入口职责重叠。
- 支付仍以 simulated escrow 为主，缺少 wallet、deposit、USDT 链上入金、人工审核、余额锁定、资金流水、付款凭证、风控审核闭环。
- Google Maps / 地区 / 区域范围能力缺失，目前只有 `country`、`city`、`lat`、`lng`、`radius_km` 字段和简单 haversine 匹配。
- H5 缺少大量移动端关键流程，包括地图半径选择、充值、USDT 支付、KYB 上传、分支覆盖范围、纠纷详情、通知设置、移动端紧急后台。
- 企业后台缺少完整详情页、审批流、权限矩阵、支付审核、证据时间线、风控处理、运营配置和审计追溯体验。
- 后台管理必须是单独的服务端和单独的前端应用，不能混在买家/卖家 PC 或 H5 前台项目中。
- 买家和卖家都缺少成交率、资料完整度、押金金额、历史行为综合计算的信誉值和等级体系。

## 2. 优先级定义

### P0 — 必须补齐

P0 是能否形成可用闭环的基础能力。

- 统一后端 OpenAPI、PC/H5/admin 类型、状态枚举和 API 路径。
- 补齐真实 wallet / deposit / escrow ledger 数据模型和接口。
- 支持 USDT deposit intent、tx hash 提交、后台人工验证、余额入账。
- 补齐 Google Maps 地址搜索、地理编码、反向地理编码、半径覆盖估算。
- H5 走通 buyer 端核心闭环：发布需求、查看报价、选择报价、付款/托管、确认收货、纠纷。
- 后台走通运营闭环：用户、公司审核、订单、托管、USDT/银行入金审核、退款、释放、风控、审计。
- 后台管理从用户端彻底分离，形成独立 admin 服务端和独立 admin 前端。
- 买家和卖家都有 `trust_score`、`trust_tier`、成交率、资料完整度、押金金额展示。

### P1 — 企业级增强

- Region / Service Area 管理后台。
- 多币种钱包、平台手续费、服务费、汇率展示。
- 供应商 payout destination、USDT wallet、Payoneer/bank/manual payout 占位与审核。
- 移动端供应商完整操作：分支管理、覆盖范围、触发规则、目录、报价、交付证明。
- 纠纷证据时间线、裁决模板、内部备注、用户可见备注。
- 员工角色权限矩阵、操作审批、敏感操作二次确认。

### P2 — 后续扩展

- 自动链上 USDT 监听和确认数同步。
- Stripe / PayMongo / Xendit / Payoneer 真实集成。
- 多地图供应商切换。
- 区域热力图、价格智能、广告推广、复杂风控模型。
- Capacitor 打包、推送通知、离线缓存。

## 3. API 与类型契约统一

### 3.1 后端 OpenAPI 为唯一事实来源

后续前端开发必须以后端 OpenAPI 为准，不允许 PC、H5、admin 各自维护互相冲突的状态枚举。

需要统一：

- `UserRole`
- `UserStatus`
- `CompanyStatus`
- `VerificationLevel`
- `IntentStatus`
- `OfferStatus`
- `OrderStatus`
- `EscrowStatus`
- `DeliveryStatus`
- `DisputeStatus`
- `NotificationStatus`
- `PaymentEventStatus`
- `PayoutStatus`
- `WalletTransactionStatus`
- `DepositStatus`

### 3.2 当前已发现的不一致

必须修正这些前端封装或页面调用：

- 订单列表应调用 `GET /orders/my`，不是 `/orders`。
- 订单确认收货应调用 `POST /orders/{order_id}/accept`，不是 `/orders/{id}/deliver`。
- 供应商目录应调用 `/supplier/catalog/items`，不是 `/catalog` 或 `/items`。
- 报价提交应调用 `POST /intents/{intent_id}/offers`，不是单独 `/offers`。
- 纠纷创建应调用 `POST /orders/{order_id}/dispute`，不是单独 `/disputes`。
- H5 和 PC 的 `IntentStatus`、`OrderStatus`、`EscrowStatus` 定义必须与后端模型一致。

### 3.3 前端 API 客户端要求

- PC、H5、admin 共享同一套生成类型或同步类型文件。
- API client 必须按业务域拆分：`auth`、`users`、`companies`、`intents`、`offers`、`orders`、`escrow`、`wallets`、`maps`、`admin`。
- 所有页面禁止直接硬编码 mock 数据作为最终实现；允许 seed/demo 数据来自后端。
- 所有敏感操作必须展示后端返回的真实错误，不能只用 `alert('Failed')`。

## 4. Google Maps / 地区与区域范围

### 4.1 后端地图服务抽象

后端新增 `maps` 模块，默认 provider 为 Google Maps，但必须保留 provider abstraction，避免未来锁死单一地图服务。

环境变量：

```env
MAP_PROVIDER=GOOGLE
GOOGLE_MAPS_API_KEY=
GOOGLE_MAPS_REGION=PH
GOOGLE_MAPS_LANGUAGE=en
MAPS_CACHE_TTL_SECONDS=86400
```

后端服务能力：

- Place Autocomplete：城市、区域、街道、商圈搜索。
- Geocoding：地址转 `lat/lng`。
- Reverse Geocoding：`lat/lng` 转地址和行政区。
- Place Details：获取 `place_id`、formatted address、viewport、行政层级。
- Coverage Estimate：根据买家位置、半径、分类、供应商分支估算覆盖数量。

### 4.2 建议 API

```text
GET /maps/autocomplete?query=&country=&language=
GET /maps/geocode?address=&country=
GET /maps/reverse-geocode?lat=&lng=&language=
GET /maps/place-details?place_id=
GET /maps/coverage/estimate?category_id=&lat=&lng=&radius_km=
```

后台区域管理：

```text
GET    /admin/regions
POST   /admin/regions
PATCH  /admin/regions/{region_id}

GET    /admin/service-areas
POST   /admin/service-areas
PATCH  /admin/service-areas/{service_area_id}
```

### 4.3 数据模型要求

新增或等价支持：

- `regions`
  - `id`
  - `country_code`
  - `admin_level`
  - `name`
  - `parent_region_id`
  - `google_place_id`
  - `center_lat`
  - `center_lng`
  - `viewport_json`
  - `status`
- `service_areas`
  - `id`
  - `owner_type`: `PLATFORM` / `COMPANY` / `BRANCH`
  - `owner_id`
  - `region_id`
  - `center_lat`
  - `center_lng`
  - `radius_km`
  - `polygon_json`
  - `coverage_type`: `RADIUS` / `POLYGON` / `REGION`
  - `status`

### 4.4 前端体验要求

Buyer 发布需求：

- 支持搜索城市/区域。
- 支持使用当前位置。
- 支持拖动地图 pin。
- 支持 1km、3km、5km、10km、25km、50km、自定义半径。
- 地图上显示圆形覆盖范围。
- 显示预计可触达供应商数量。

Supplier 分支管理：

- 每个分支必须能设置服务地址、经纬度、服务半径或区域。
- 可预览覆盖范围。
- 可按分类或库存能力设置不同覆盖规则。

Admin 区域管理：

- 可维护 Cebu、Mandaue、Lapu-Lapu 等重点区域。
- 可启停区域、设置默认半径、查看区域内活跃买家需求和供应商覆盖。
- Dashboard 可显示 region heatmap placeholder，P1 再做真实热力图。

## 5. Wallet / Deposit / USDT / Payment

### 5.1 当前问题

现有后端已有 `EscrowTransaction`、`PaymentEvent`、`Payout`，但缺少完整资金账户体系：

- 买家没有 wallet balance。
- 没有 deposit intent。
- 没有 USDT 网络、充值地址、tx hash、确认数。
- 没有钱包入账和余额锁定。
- 没有统一 ledger 记录所有资金变动。
- 人工验证只挂在 escrow 上，不足以支撑充值、余额、退款、风控审计。

### 5.2 钱包能力

Buyer wallet：

- 查看可用余额。
- 查看锁定余额。
- 查看充值记录。
- 查看托管锁定记录。
- 查看退款入账。
- 使用余额支付订单托管。

Supplier wallet / payout：

- 查看待释放金额。
- 查看已释放金额。
- 设置 payout destination。
- 支持 bank、Payoneer、USDT wallet 占位。

### 5.3 USDT deposit 流程

Buyer 创建充值：

1. 选择币种 `USDT`。
2. 选择网络：`TRC20`、`ERC20`、`BEP20`，V1 推荐先支持 `TRC20` 占位。
3. 输入或选择充值金额。
4. 后端创建 `deposit_intent`，返回平台收款地址、memo/tag 如有、过期时间。
5. 用户完成链上转账后提交 `tx_hash`。
6. 后台 finance officer 审核 tx_hash、金额、网络、截图或链上记录。
7. 审核通过后 wallet 入账，并写入 ledger、payment event、audit log。
8. 审核拒绝后进入 `REJECTED`，记录原因并通知用户。

状态：

```text
DepositStatus:
PENDING_PAYMENT
TX_SUBMITTED
UNDER_REVIEW
CONFIRMED
CREDITED
REJECTED
EXPIRED
CANCELED
```

### 5.5 数据模型要求

新增或等价支持：

- `wallets`
  - `id`
  - `user_id`
  - `company_id`
  - `currency`
  - `available_balance_minor`
  - `locked_balance_minor`
  - `status`

### 5.6 后台支付审核

Admin finance 模块必须支持：

- USDT 待审核列表。
- 银行转账待审核列表。
- 充值详情页，展示用户、金额、网络、地址、tx hash、截图、风险标记。
- 审核通过、拒绝、请求更多信息。
- 通过后自动 wallet 入账。
- 拒绝后记录用户可见原因。
- 所有操作写 audit log，risk level 为 `HIGH` 或 `CRITICAL`。

## 6. H5 移动端缺失功能补全

H5 不能只是桌面端压缩版，必须是移动优先的完整关键流程。

### 6.1 Buyer H5

必须补齐：

- 钱包首页：余额、锁定金额、充值按钮、交易记录。
- USDT deposit：选择网络、查看充值地址、复制地址、上传截图、提交 tx hash、查看审核状态。
- 发布需求地图步骤：搜索地址、GPS 定位、地图 pin、半径圈、供应商覆盖估算。
- 支付/托管步骤：余额支付、USDT deposit 引导、manual bank 占位、托管状态说明。
- 订单详情：托管状态、交付状态、确认收货、发起纠纷。
- 纠纷详情：证据列表、补充证据、管理员裁决、退款/释放结果。
- 通知设置：站内、Email、Telegram。
- 个人/公司资料、认证状态。

### 6.2 Supplier H5

必须补齐：

- 分支管理：地址、地图定位、服务半径。
- 覆盖范围预览：当前分支能收到哪些区域需求。
- 匹配触发规则：分类、区域、预算范围、库存关键词。
- KYB 上传：公司文件、证件、状态反馈。
- payout settings：bank、Payoneer、USDT wallet 占位。
- 订单交付证明：拍照/上传、tracking number、状态更新。
- 报价详情、撤回报价、订单详情、消息。

### 6.3 Admin Emergency H5

H5 后台只做紧急操作，不替代 PC 后台：

- Dashboard 概览：待审核、争议、支付异常、风险预警。
- 公司审核 quick review。
- USDT/manual bank deposit quick review。
- 纠纷 quick review。
- 风险详情和标记处理。
- Escrow transaction 状态查看，不默认开放大额释放/退款，除非角色和二次确认满足。

## 7. 企业级后台补全

### 7.1 后台必须独立部署

后台管理必须作为独立系统拆出去，不能混在买家/卖家前台里。

目标结构：

```text
admin-backend/                 # 独立后台管理服务端，或 backend 内独立 admin app 部署单元
├── app/
│   ├── main.py
│   ├── routers/
│   ├── services/
│   ├── schemas/
│   └── models/
admin-frontend/                # 独立后台管理前端
buyer/supplier frontend        # PC/H5 用户端，不包含后台管理页面
```

拆分要求：

- `admin-frontend` 作为唯一真实后台管理前端。
- 后台 API 使用独立 admin 服务端入口，推荐独立域名：`admin-api.{domain}`。
- 后台前端使用独立域名：`admin.{domain}`。
- 用户端 PC/H5 不再包含 `/admin/*` 核心后台页面。
- `pc-frontend/pages/admin/*` 必须迁移到 `admin-frontend`，迁移完成后删除或只保留跳转提示。
- 后台认证、员工角色、权限、审计、敏感操作二次确认必须独立于买家/卖家用户端。
- 后台服务端可以复用主业务数据库，但必须有独立配置、独立 CORS、独立路由前缀、独立部署和独立访问控制。
- 不允许长期存在两套后台、两套权限、两套接口封装。

后台服务端职责：

- Staff / Admin 登录。
- RBAC 权限校验。
- 用户、公司、订单、支付、钱包、风控、纠纷、审计管理。
- 系统配置和第三方集成配置。
- 高风险操作审批和审计日志。

用户端职责：

- Buyer 发布需求、支付、订单、纠纷、钱包。
- Supplier 报价、目录、分支、交付、提现。
- 不暴露后台管理能力。

### 7.2 Admin 必补页面

用户与权限：

- 用户状态变更。
- 员工邀请。

公司与审核：

订单与交易：

- Escrow 详情。
- Wallet ledger 详情。
- Payment events。

纠纷：

系统配置：

## 9. 安全、权限与审计

### 9.1 权限

- 普通用户不能直接修改 escrow/payment/wallet 状态。
- Buyer 只能查看自己的 wallet、deposit、order、dispute。
- Supplier 只能查看自己公司相关订单、报价、payout。
- Finance Officer 可处理 deposit、escrow、payout。
- Risk Analyst 可标记风险，但不能释放资金。
- Super Admin 可做最高风险操作，但必须记录 reason。

### 9.2 审计

以下操作必须写 audit log：

- USDT deposit 审核通过/拒绝。
- 钱包余额调整。
- Escrow release/refund。

## 10. 买家/卖家信誉值与等级体系

平台必须为买家和卖家都建立信誉值，不只做供应商认证。信誉值用于排序、曝光、报价可信度、押金要求、风控提醒和后台审核优先级。

### 10.1 核心字段

用户和公司维度都需要保存信誉相关字段：

```text
trust_score                 # 0-1000 综合信誉分
trust_tier                  # BRONZE / SILVER / GOLD / PLATINUM / DIAMOND
profile_completion_rate     # 资料完整度 0-100
deal_completion_rate        # 成交率 0-100
deposit_amount_minor        # 已缴押金金额
deposit_currency            # 押金币种
verified_deposit_minor      # 已审核通过押金金额
successful_deals_count      # 成功成交数
canceled_deals_count        # 取消数
dispute_rate                # 纠纷率
refund_rate                 # 退款率
late_delivery_rate          # 卖家迟交率
late_payment_rate           # 买家迟付率
last_score_calculated_at
```

买家信誉重点：

- 发布需求后是否真实成交。
- Award offer 后是否按时付款或托管。
- 是否频繁取消订单。
- 是否恶意纠纷。
- 资料是否完整。
- 是否缴纳押金。
- 历史成交金额和成交次数。

卖家信誉重点：

- 报价后是否履约。
- 是否按时交付。
- 交付后确认率。
- 纠纷率和退款率。
- 公司/KYB 资料完整度。
- 是否缴纳供应商押金。
- 分支、目录、联系方式、营业资料是否完整。

### 10.2 信誉值计算建议

V1.1 可先使用可解释规则模型，不需要 AI 风控。

```text
trust_score =
  profile_score * 0.20
  + deal_score * 0.35
  + deposit_score * 0.20
  + verification_score * 0.15
  + risk_penalty_score * 0.10
```

评分说明：

- `profile_score`：资料完整度，姓名/公司名/手机号/邮箱/地址/头像/KYB/分支/目录越完整越高。
- `deal_score`：成交率、成功成交次数、成交金额、取消率、履约率综合。
- `deposit_score`：押金金额越高越可信，但必须设置上限，不能无限买分。
- `verification_score`：邮箱、手机、公司、KYB、管理员审核等级。
- `risk_penalty_score`：纠纷、退款、违规、风控标记、站外交易会扣分。

押金估算必须区分买家和卖家：

- 买家押金用于证明采购诚意，可影响供应商是否优先报价。
- 卖家押金用于证明履约能力，可影响买家选择报价时的信任判断。
- 押金不是付款，不等同 escrow，可退还、冻结、扣罚或用于风险保证金。

### 10.3 等级规则

默认等级：

```text
0-199     BRONZE
200-399   SILVER
400-699   GOLD
700-849   PLATINUM
850-1000  DIAMOND
```

中文展示：

- `BRONZE`：青铜
- `SILVER`：白银
- `GOLD`：黄金
- `PLATINUM`：铂金
- `DIAMOND`：钻石

等级展示要求：

- Buyer request card 显示买家信誉等级和押金状态。
- Supplier offer card 显示卖家信誉等级、成交率、押金状态、KYB 状态。
- Admin 用户详情和公司详情显示评分拆解。
- H5 端必须有紧凑 badge，避免占据太多移动端空间。
- 后台可手动调整或冻结信誉分，但必须写 audit log。

### 14.4 广告投放 / 排名

#### 14.4.1 商户端广告能力

商户可购买广告排名提升商品/店铺曝光：

- 选择推广对象：单个商品、整个店铺、特定分类。
- 选择投放分类和区域。
- 选择广告包：按天数计费（如 7天/30天/90天），初版固定价格，不做实时竞价。
- 设置预算和起止日期。
- 查看广告效果：曝光数、点击数、询盘数、订单转化数。

#### 14.4.2 后台广告管理

后台可对广告进行全生命周期管理：

- 审核：新提交的广告需人工或自动审核。
- 暂停/恢复：违规或投诉时可暂停。
- 拒绝：内容违规、虚假信息可拒绝并退款。
- 手动加权：运营可给优质商户额外排名权重。
- 配置推荐位：首页 banner、分类页置顶、搜索页优先。

#### 14.4.3 广告展示要求

- 广告内容必须明确展示 `Sponsored` / `广告` 标签，不允许隐藏广告身份。
- 广告卡片在视觉上与自然结果保持一致，仅多出 Sponsored 角标。
- 广告不得展示虚假价格、虚假库存、虚假信誉。

#### 14.4.4 广告优先级

- P0：商户购买 N 天广告包，后台 CRUD、审核、启停，feed 中标记广告。
- P2：CPC/CPM 计费、实时竞价排名、自动出价优化。

### 14.5 推荐与排名规则

#### 14.5.1 排名因素

```text
ranking_score =
  ad_weight           * W1    # 广告权重（有付费广告时加分）
  + category_match    * W2    # 分类匹配度
  + distance_score    * W3    # 距离越近分越高
  + trust_score       * W4    # 信誉值
  + deal_completion   * W5    # 成交率
  + profile_complete  * W6    # 资料完整度
  + deposit_score     * W7    # 押金金额
  + response_speed    * W8    # 响应速度
  + stock_available   * W9    # 库存/可用性
  + freshness         * W10   # 发布/更新时间新鲜度
```

#### 14.5.2 排名规则配置

后台可配置不同页面使用不同的 ranking rule：

```text
- 首页推荐：ad_weight 高、trust_score 高、distance 中
- 分类页：category_match 高、ad_weight 中、distance 中
- 区域页：distance 高、category_match 中、trust_score 中
- 搜索页：text_relevance 高、ad_weight 中、trust_score 中
```

支持针对用户类型（buyer/consumer）、地区、品类、语言、币种、B2B/B2C 模式做定向排序规则。

### 14.6 B2B + B2C 双模式

#### 14.6.1 模式定义


- 商户创建 listing 时选择模式。
- 分类和 feed 筛选支持按模式过滤。

#### 14.6.2 模式差异

| 特性 | B2B | B2C |
|------|-----|-----|
| 定价 | 批发价、MOQ、阶梯价 | 零售价、规格选择 |
| 流程 | RFQ → 报价 → 比价 → 托管 | 立即购买 / 加购物车 |
| 信誉重点 | 成交率、履约率、KYB | 好评率、退货率 |
| CTA | "Request Quote" / "发布需求" | "Buy Now" / "立即购买" |
| 支付 | 托管 + 余额 + USDT | 托管 + 余额 + USDT + 在线支付占位 |

#### 14.6.3 B2C 实施节奏

- P0：`market_mode` 字段、listing 和 feed 支持模式标签和筛选。
- P1：B2C 立即购买链路、库存扣减、规格选择、配送/自提占位。
- P2：购物车、合并支付、售后退货。

**P1：**

**P2：**

## 15. 收发货地址管理

> 当前系统面向全球用户，买家和供应商分布在不同国家/城市。系统需要完整的地址簿管理，并区分买家收货地址与供应商发货地址。

### 15.1 当前差距

- `User` 模型没有任何地址字段。
- `Company` 有 country/city/address 但不是结构化地址簿。
- `Branch` 有 country/city/address/lat/lng 但这是服务范围，不是发货仓库地址。
- `Intent` 有 country/city/lat/lng 但只是需求地点，不是真正的收货地址。
- 买家没有地址簿，无法管理多个收货地址。
- 供应商没有独立的发货仓库地址，下单后不知道从哪里发货。
- 没有标准化国际地址格式（国家、省/州、城市、区、邮编、详细地址、联系人、电话）。

### 15.2 地址模型

```text
Address {
  id: UUID
  user_id: UUID
  company_id: UUID | null
  address_type: SHIPPING_FROM | DELIVERY_TO | WAREHOUSE | BILLING
  label: string             # "我的公司仓库" / "Home" / "Office"
  contact_name: string      # 收件人/发件人
  contact_phone: string
  country_code: string      # ISO 3166-1 alpha-2 (PH, CN, US, JP ...)
  country_name: string
  state_province: string    # 省/州/大区
  city: string
  district: string | null   # 区/县
  postal_code: string
  address_line1: string     # 街道/门牌/大厦
  address_line2: string | null
  lat: float | null
  lng: float | null
  google_place_id: string | null
  is_default: boolean
  status: ACTIVE | DELETED
  created_at, updated_at
}
```

地址类型说明：

- `SHIPPING_FROM`：供应商发货地址（从哪里发）。
- `DELIVERY_TO`：买家收货地址（送到哪里）。
- `WAREHOUSE`：供应商仓库地址。
- `BILLING`：账单地址（可选）。

### 15.3 买家地址管理

买家可以：

- 添加多个收货地址（`DELIVERY_TO`）。
- 设置默认收货地址。
- 编辑/删除地址。
- 发布需求时选择收货地址或新增。
- 下单时确认或更换收货地址。
- 地址搜索支持 Google Maps 自动补全。

### 15.4 供应商地址管理

供应商可以：

- 添加多个发货/仓库地址（`SHIPPING_FROM` / `WAREHOUSE`）。
- 每个分支绑定一个发货地址。
- 报价时选择从哪个仓库/分支发货。
- 报价中的物流费用基于发货地址与买家收货地址自动计算。

### 15.5 建议 API

```text

POST   /addresses                    # 新增地址
PATCH  /addresses/{address_id}       # 修改地址
DELETE /addresses/{address_id}       # 删除地址（软删除）
POST   /addresses/{address_id}/set-default  # 设为默认
```

### 15.6 前端页面

- PC Buyer Settings：地址簿管理标签页。
- PC Supplier Settings：发货地址/仓库地址管理。
- H5 Profile：地址管理入口。
- 发布需求流程：选择收货地址步骤。
- 报价流程：选择发货地址步骤。
- 订单详情：展示收货地址和发货地址。

## 16. 国际物流计算系统

> 系统面向全球用户，买卖双方可能在不同国家，需要支持国际物流费用预估，让买家在比价时能看到含物流费的总价，做出更合理的采购决策。

### 16.1 当前差距

- `Offer` 有 `delivery_fee_minor` 但完全由供应商手动填写，没有任何物流计算逻辑。
- 没有物流方式、物流路线、物流报价模型。
- 没有重量/体积维度数据。
- 买家看到的价格不包含物流预估，无法评估跨境总成本。
- 后台没有物流费率配置能力。

### 16.2 物流方式

```text
ShippingMethod:
  SEA_FREIGHT       # 海运（最便宜，最慢，适合大宗）
  AIR_FREIGHT       # 空运（较贵，较快）
  EXPRESS           # 国际快递（最贵，最快，适合小件样品）
  LAND_FREIGHT      # 陆运（适合邻国/大陆内）
  LOCAL_DELIVERY    # 本地配送（同城/同区域）
  SELF_PICKUP       # 自提
```

### 16.3 物流费率模型

```text
ShippingRoute {
  id: UUID
  origin_country: string          # 发货国家 ISO code
  origin_region: string | null    # 发货区域（可选细分）
  dest_country: string            # 目的国家
  dest_region: string | null      # 目的区域
  shipping_method: ShippingMethod
  status: ACTIVE | INACTIVE
  created_at, updated_at
}

ShippingRate {
  id: UUID
  route_id: UUID                  # 关联 ShippingRoute
  weight_min_kg: float            # 最小重量（含）
  weight_max_kg: float            # 最大重量（含）
  price_per_kg_minor: int         # 每公斤价格（最小货币单位）
  currency: string
  min_charge_minor: int           # 最低收费
  volume_factor: float            # 体积重换算系数（默认 5000 for 空运, 1000 for 海运）
  estimated_days_min: int         # 预估最短天数
  estimated_days_max: int         # 预估最长天数
  surcharges_json: JSON | null    # 附加费：燃油费、旺季附加费、偏远地区等
  valid_from: date
  valid_until: date | null
  notes: string | null
  status: ACTIVE | INACTIVE
  created_at, updated_at
}
```

### 16.4 物流计算逻辑

计算输入：

```text
origin_country, origin_region        # 发货地（供应商地址）
dest_country, dest_region            # 收货地（买家地址）
shipping_method                      # 物流方式
weight_kg: float                     # 实际重量
length_cm, width_cm, height_cm       # 外箱尺寸（可选，用于体积重）
qty: int                             # 数量
declared_value_minor: int            # 申报价值（用于保险费计算）
```

计算输出：

```text
ShippingEstimate {
  route_id: UUID
  shipping_method: ShippingMethod
  actual_weight_kg: float
  volumetric_weight_kg: float | null
  chargeable_weight_kg: float        # max(实际重量, 体积重量)
  base_cost_minor: int               # 基础运费
  surcharges: [{ name, amount_minor }]
  insurance_minor: int | null
  customs_estimate_minor: int | null  # 关税预估（可选）
  total_shipping_minor: int           # 总运费
  currency: string
  estimated_days_min: int
  estimated_days_max: int
}
```

体积重量计算：

```text
空运: volumetric_kg = (L × W × H) / 5000
海运: volumetric_kg = (L × W × H) / 1000
快递: volumetric_kg = (L × W × H) / 5000
计费重量 = max(actual_kg, volumetric_kg)
```

### 16.5 买卖家视角

#### 买家看到的价格

在报价列表/比价页面，每个报价展示：

```text

商品小计:        ₱60,000.00
```

## 18. 个人/企业账户、游客 Marketplace 与智能匹配排名细化

> 本章是对第 14 章（Marketplace/广告/B2B/B2C）的细化补充，新增个人与企业账户区分、游客浏览体验、登录回跳、Buyer Dashboard 返回入口、以及需求发布后的智能候选卖家排名机制。

---

### 18.1 个人/企业账户区分（AccountType）

#### 18.1.1 需求背景

当前所有 BUYER 和 SUPPLIER_ADMIN 角色无法区分"个人"和"企业"身份。买卖双方都需要在注册时选择账户类型，以确定：
- 是否需要填写企业资料（KYB）。
- 是否显示"团队管理"入口。
- 商品/服务的定价模式（零售 vs 批发）。
- Marketplace 筛选中的商户类型。

#### 18.1.2 AccountType 枚举

不新增角色，在现有 `BUYER` / `SUPPLIER_ADMIN` 上通过 `account_type` 字段区分：

```text
AccountType: INDIVIDUAL | BUSINESS
```

加到 `users` 表：

```text
User {
  ...existing fields...
  account_type: AccountType   # INDIVIDUAL | BUSINESS，注册时必填
}
```

#### 18.1.3 注册流程差异

**买家注册：**

- 选择账户类型：`个人买家（Individual）` 或 `企业买家（Business）`。
- `INDIVIDUAL` 买家：
  - 只需填写姓名、邮箱、手机、密码。
  - 不需要填写企业资料，不显示 KYB 入口。
  - Dashboard 不显示"公司资料"、"团队管理"标签。
  - 适合 B2C 场景：直接购买商品或发布小额采购需求。
- `BUSINESS` 买家：
  - 除个人信息外，需填写公司名称、行业、预计采购规模。
  - 显示 KYB 上传入口（营业执照、税号等）。
  - Dashboard 显示"公司资料"、"团队"入口。
  - 适合 B2B 场景：大批量采购、报价比价、托管。

**卖家注册：**

- 选择账户类型：`个人卖家（Individual）` 或 `企业卖家（Business）`。
- `INDIVIDUAL` 卖家：
  - 偏向 B2C，适合个人手工、小批量零售、服务类。
  - 不强制 KYB，但鼓励上传身份证提升信誉。
  - 列表默认 `market_mode = B2C`，可手动改为 `BOTH`。
  - 不需要"分支管理"入口（单人单点经营）。
- `BUSINESS` 卖家：
  - 支持 B2B / B2C / BOTH 模式。
  - 需要 KYB（营业执照、税号、银行账户）。
  - 可以创建多个分支和仓库地址。
  - 显示"团队管理"和员工邀请功能。

#### 18.1.4 建议 API

```text
POST /auth/register            # 注册时携带 account_type
PATCH /auth/me/account-type    # 升级个人为企业（不允许降级）
GET  /users/me/account-context # 返回当前用户的账户类型上下文、可用功能列表
```

`account-context` 响应示例：

```json
{
  "account_type": "INDIVIDUAL",
  "role": "BUYER",
  "features": {
    "kyb": false,
    "company_profile": false,
    "team_management": false,
    "b2b_rfq": true,
    "b2c_buy_now": true
  }
}
```

#### 18.1.5 数据模型

```text
AccountType: INDIVIDUAL | BUSINESS

ALTER users ADD COLUMN account_type VARCHAR(20) DEFAULT 'INDIVIDUAL'
```

迁移要求：
- 存量用户默认 `INDIVIDUAL`，可后台升级为 `BUSINESS`。
- `account_type` 写入 audit log 当发生变更时。

#### 18.1.6 前端适配

PC 前端（Buyer Dashboard）：
- `account_type === INDIVIDUAL`：隐藏"公司资料"、"团队"、"KYB"标签，只显示个人设置。
- `account_type === BUSINESS`：完整显示全部功能标签。

H5 前端（Profile）：
- 同上，根据 `account_type` 动态显示/隐藏功能入口。

Admin 后台：
- 用户详情新增 `Account Type` 字段。
- 筛选支持：按 `INDIVIDUAL` / `BUSINESS` 过滤用户列表。

### 18.2 游客访问 Marketplace（Public Feed）

#### 18.2.1 需求背景

当前 marketplace feed 可能被登录中间件拦截，导致游客无法浏览商品。根据标准电商产品体验，marketplace 必须对游客完全开放，只有发起交易动作时才要求登录。

#### 18.2.2 游客可访问的页面

```text
GET /marketplace              # PC 瀑布流首页
GET /marketplace/{item_id}    # 商品/服务详情页
GET /categories               # 分类浏览页
GET /categories/{id}/feed     # 分类瀑布流
```

这些页面**不得要求登录**，即不能有 `auth` middleware 阻断游客访问。

#### 18.2.3 仅登录后才允许的操作

以下操作触发登录弹窗（modal）或跳转到 `/login?return_url=当前页面`：

```text
- 发布需求 / Post Request
- 提交报价 / Make Offer
- Request Quote（B2B）
- Buy Now（B2C）
- 收藏商品 / 关注商户
- 发起消息 / 联系卖家
- 下单 / 支付
```

#### 18.2.4 登录后回跳机制

- 所有触发登录的场景必须携带 `return_url` 参数。
- 登录/注册成功后，自动回跳到 `return_url` 指定的页面。
- `return_url` 必须保留原始筛选条件（query string），不能只回跳到首页。

示例流程：

```text
游客 → 点击 "Request Quote"
→ 跳转 /login?return_url=/marketplace?category=construction&city=cebu
→ 登录成功
→ 自动回跳 /marketplace?category=construction&city=cebu
→ 弹出 Request Quote 表单（可选：通过 session 记住操作意图）
```

#### 18.2.5 建议 API

```text
GET  /marketplace/feed          # PUBLIC，无需 token
GET  /marketplace/items/{id}    # PUBLIC，商品详情，无需 token
POST /marketplace/actions/require-auth-return  # 记录操作意图，登录后自动触发
```

注意：`/marketplace/feed` 和 `/marketplace/items/{id}` 后端路由**不得加** `get_current_user` 依赖，改为可选认证（`Optional[User]`），有 token 时个性化推荐，无 token 时返回通用结果。

#### 18.2.6 前端适配

PC 前端：
- `marketplace.vue`：移除或降级 auth middleware，改为 public 访问。
- 交易按钮（Request Quote / Buy Now）点击时检查登录状态，未登录弹出登录 modal 或跳转。
- 登录 modal 关闭后，自动继续之前的操作（如打开报价表单）。

H5 前端：
- 同上，底部导航的 marketplace 图标不得被登录 middleware 拦截。

### 18.5 需求发布后自动关联卖家与智能候选排名

#### 18.5.1 需求背景

当前买家发布需求后，系统仅通过 `find_matching_suppliers` 服务发送站内通知，但没有生成"候选卖家排名列表"供买家主动选择或系统展示。买家只能被动等待供应商报价，缺少"系统推荐这些供应商"的主动展示能力。

#### 18.5.2 候选卖家排名流程

```text
买家发布需求
    ↓
后端 matching_service 执行多维度匹配
    ↓
生成 candidate_supplier_list（含分数和 score breakdown）
    ↓
存入 intent_candidates 表（或缓存，视并发量决定）
    ↓
买家可在需求详情页查看"推荐供应商"列表
    ↓
供应商收到 Ping 后提交报价
    ↓
报价列表 + 候选列表融合展示（有报价的候选排前面）
```

#### 18.5.3 排名维度

```text
ranking_score =
  category_match_score    * W1   # 分类和属性匹配度（关键词/品牌/型号/规格）
  + distance_score        * W2   # 供应商分支与买家交货地点的距离
  + shipping_cost_score   * W3   # 预估物流成本（越低越高分）
  + trust_score           * W4   # 供应商综合信誉值
  + deal_completion_rate  * W5   # 成交率（历史按时履约比例）
  + response_speed_score  * W6   # 平均响应速度（历史报价响应时间）
  + deposit_score         * W7   # 押金金额（提升可信度）
  + stock_availability    * W8   # 是否有相关库存
  + delivery_speed_score  * W9   # 预估交期
  + ad_boost_score        * W10  # 广告权重（商户购买了广告加成）
```

#### 18.5.4 排序选项（买家端）

买家可选择排序方式：

```text
- 综合最优（默认）：综合所有维度权重
- 成本最低：侧重 shipping_cost + 报价金额（有报价时）
- 信誉最高：侧重 trust_score + deal_completion_rate
- 距离最近：侧重 distance_score
- 交期最快：侧重 delivery_speed_score
```

#### 18.5.5 Score Breakdown 展示

每个候选卖家的推荐理由必须可解释：

```json
{
  "supplier_id": "...",
  "supplier_name": "Cebu Supply Co.",
  "ranking_score": 847,
  "score_breakdown": {
    "category_match": 95,
    "distance_km": 3.2,
    "shipping_cost_est": 450,
    "trust_tier": "GOLD",
    "trust_score": 720,
    "deal_completion_rate": 0.92,
    "has_stock": true,
    "delivery_days_est": 2,
    "is_sponsored": false
  },
  "why_recommended": "距离 3.2km · 成交率 92% · 有现货 · 信誉黄金"
}
```

#### 18.5.6 后台排名权重配置

后台运营可配置不同场景的排名权重：

```text
RankingProfile {
  id
  name: string                  # "默认" / "B2C优先" / "价格优先"
  context: INTENT_MATCHING | MARKETPLACE_FEED | SEARCH
  weights_json: {
    category_match: 0.25,
    distance: 0.20,
    trust_score: 0.20,
    deal_completion: 0.15,
    shipping_cost: 0.10,
    response_speed: 0.05,
    ad_boost: 0.05
  }
  is_default: boolean
  status: ACTIVE | INACTIVE
}
```

#### 18.5.7 建议 API

用户端：

```text

     # query: sort=comprehensive|cost|trust|distance|delivery&page=&limit=

POST /intents/{intent_id}/ranking-preferences
     # 买家设置个人偏好排序方式，影响候选列表排序

```

```text

POST   /ranking/profiles               # 新建排名配置
PATCH  /ranking/profiles/{id}          # 修改权重
GET    /admin/ranking-rules            # 与 marketplace feed 排名规则共用
POST   /admin/ranking-rules
PATCH  /admin/ranking-rules/{rule_id}
```

  - 本地数据库使用 compose db 时宿主机端口是 `5433`。
- 生产部署文档必须明确：
  - 生产 compose 只应自动迁移，不应默认运行本地 seed。
  - 若执行 seed，必须先确认是否允许写入测试/demo 数据。
  - 生产必须关闭 `DEMO_MODE`。

### 20.4 验收标准

- Demo Mode 开启：demo buyer/supplier 可登录，H5/PC 可展示 demo 辅助数据。
- Demo Mode 关闭：demo buyer/supplier 登录失败，H5/PC 不再自行展示本地模拟数据。
- 买家需求详情页 PC/H5 都能打开候选供应商详情。
- 买家能绑定候选供应商，刷新后该候选显示 bound 状态。
- 供应商收到买家绑定候选的站内通知。
- 其他 AI agent 读取 README / DEPLOY_SKILL / deploy README 后，不会把生产部署目录误认为本地开发入口。

## 21. AI Project Forge：买家项目需求分析、资料收集与智能采购清单

### 21.1 背景

当前平台已经具备 marketplace、买家需求 `Intent`、供应商匹配、报价、订单、支付与物流的基础链路。但买家在真实场景里经常不是只买一个单品，而是有一个完整项目目标：

- 建一栋别墅、厂房、仓库、办公室或小型工厂。
- 做光伏项目，需要根据屋顶、功率、预算推算光伏板、逆变器、线缆、支架、电池等。
- 做科技/极客项目，需要开发板、电源、传感器、外壳、线材、工具等组合清单。
- 做装修、园区、农业、维修、改造等项目，需要 AI 先帮助拆解资源和采购项。

因此需要新增 buyer 侧 `AI Project Forge` 能力：用户可以输入基础描述、地点、面积、用途、预算、质量偏好，并上传 PDF、图片、TXT、DOCX 等资料。后台 AI provider 对资料做抽取、追问、分析和估算，生成可编辑的项目采购清单。买家确认后，系统再把项目分项拆成普通采购需求，复用现有供应商匹配、报价、成交、支付和物流流程。

该能力的目标不是替代专业工程师、预算师或采购经理，而是帮助买家快速把模糊项目变成结构化需求单，提高询价和匹配效率。

### 21.2 产品形态

P0 需要同时覆盖 PC 与 H5，但形态不同：

- PC 端新增类似 Idea Forge 的工作台：
  - 顶部或侧边显示步骤：项目定位、业务场景、资料上传、数据/物料模型、匹配与报价、交付计划、验收标准。
  - 中间为聊天式信息收集、结构化表单、文件上传与 AI 追问。
  - 右侧为 `Project Preview`，实时展示核心需求、物料清单、预算区间、风险提示、匹配状态和验收标准。
- H5 端新增移动优先项目创建向导：
  - 分步填写项目类型、地点、规模、预算、质量档位、描述。
  - 支持拍照/上传图片、PDF、TXT、DOCX。
  - AI 分析完成后用卡片方式展示 line items，支持编辑数量、单位、规格、质量档位。
- Admin 后台新增 AI 项目分析配置与审计：
  - 是否启用项目分析。
  - 是否启用多模态分析。
  - 项目分析默认模型。
  - prompt version。
  - AI run 状态、失败原因、原始输出与结构化输出审计。

### 21.3 P0 核心流程

P0 采用“项目总单 + 分项采购需求”模式：

1. 买家创建项目。
2. 买家填写基础信息：
   - 项目类型：`CONSTRUCTION` / `SOLAR` / `TECH_BUILD` / `RENOVATION` / `GENERAL`
   - 项目地点、城市、国家、可选地图定位。
   - 面积、数量、规模、功率或其他关键参数。
   - 预算范围、期望交付时间。
   - 质量偏好：`BUDGET` / `MID_RANGE` / `PREMIUM` / `NOT_SURE`
   - 文字描述。
3. 买家上传资料：
   - 图片：现场图、设计图、材料图、草图、设备照片。
   - PDF：图纸、需求文档、规格书、报价单。
   - TXT/DOCX：文字说明、已有清单、旧报价。
4. 后台异步启动 AI 分析。
5. AI 输出结构化结果：
   - 项目摘要。
   - 缺失问题。
   - 假设条件。
   - 推荐质量档位。
   - 物料/组件清单。
   - 预算估算。
   - 分类映射。
   - 供应商匹配建议。
   - 风险提示和验收建议。
6. 买家编辑并确认 line items。
7. 买家点击发布采购需求。
8. 系统为 confirmed line items 生成普通 `Intent`。
9. 后续供应商匹配、报价、绑定、成交、支付、物流全部复用现有链路。

P0 不做项目级合并支付。项目内每个成交报价仍按现有订单与支付流程处理，后续 P1/P2 再考虑项目级合并结算、拆单、退款与财务对账。

### 21.6 AI 工作流要求

P0 使用后台异步任务，不要求立即引入 LangGraph 依赖，但代码结构需要支持 LangGraph 风格节点拆分，方便后续替换：

- `intake_normalize`：合并用户描述、表单字段、上传文件提取文本。
- `file_extract`：PDF/TXT/DOCX 抽文本；图片在多模态开启时生成视觉摘要。
- `gap_check`：识别缺失信息并生成追问。
- `domain_plan`：判断项目类型，选择 construction/solar/tech/general 估算模板。
- `quantity_estimate`：生成物料、组件、数量、单位和规格。
- `quality_budget_estimate`：根据质量档位和预算范围做估算。
- `category_mapping`：映射到平台现有 category。
- `supplier_match_preview`：调用现有匹配逻辑估算可匹配供应商数量。
- `result_validate`：校验 JSON schema、数量、单位、必填字段。

如果 AI 输出无法通过 schema 校验，必须标记 run 为 `FAILED` 或 `NEEDS_REVIEW`，不允许生成采购需求。

### 21.7 文件上传与多模态要求

当前上传能力已有图片/PDF 基础，P0 需要补齐：

- `text/plain`
- `application/vnd.openxmlformats-officedocument.wordprocessingml.document`

文件处理规则：

- PDF/TXT/DOCX 优先抽取文本。
- 图片在 `ai_multimodal_enabled=true` 时交给多模态 provider 生成摘要。
- 图片在多模态关闭时只作为附件保存，并提示用户需要补充文字描述。
- 所有提取文本、视觉摘要和 AI 输入快照必须落库，便于审计和重试。

### 21.10 PC/H5 验收标准

- Buyer 可在 PC 创建项目并上传 PDF、图片、TXT、DOCX。
- Buyer 可在 H5 创建项目并上传手机图片或文档。
- AI 分析过程中显示 `ANALYZING` 状态，不阻塞页面。
- AI 完成后展示项目摘要、缺失问题、物料清单、预算估算和风险提示。
- 用户可编辑 line items 的名称、数量、单位、规格、质量档位和预算。
- 用户确认后可发布为普通采购需求。
- 发布后每个 line item 关联生成的 `Intent`，并能查看匹配供应商。
- PC/H5 查看同一个 project，数据完全一致。
- 后台关闭 AI 项目分析后，前端隐藏入口，API 返回明确 disabled 错误。
- AI 失败时保留草稿、上传文件、失败原因和重试入口。
